Consent, escrow, recall & disputes
for payments made by AI agents.

Tvist is one HTTP API. Before an agent pays, four checks run in a fixed order: 1) both parties agree the governing jurisdiction, 2) the payment is checked against the principal's recorded consent, 3) funds are held in escrow until delivery is proven, 4) a settled payment can be recalled only where the jurisdiction allows it and the consent was breached. Each check is one endpoint. Every button on this page sends a real request to the live service.

Executive summary — the one-minute version

Problem, market scope, solution

1 · The problem

Agents pay on rails that cannot be reversed

AI agents now spend money on people's behalf: about 1 in 6 Black Friday 2025 purchases were AI-assisted, and generative-AI retail traffic grew 4,700% year over year (Adobe Analytics). Juniper Research (2026) names trust the #1 adoption barrier for agentic commerce. The rails agents pay on — Pix, SEPA Instant, FedNow, UPI — settle in seconds and are final; the ECB puts fraud risk on instant payments at "up to 10× higher". Disputes were already expensive on cards: 106M disputes at Visa in 2025, 40–80% of e-commerce disputes classed as friendly fraud, and roughly $28B/yr in merchant chargeback losses (industry figures, compiled with sources in the executive summary below).

2 · The market scope

Existing products each cover one part

Visa, Mastercard, and Amex dispute products work on card rails only. Agent-payment protocols (Google AP2, OpenAI ACP, OKX APP, Coinbase x402) execute checkout but ship no dispute layer. Chargeback SaaS automates card representment only. Custody providers hold funds but offer no programmable, delivery-gated escrow. No product covers disputes and escrow across instant account-to-account rails — that gap is Tvist's scope.

3 · The solution

Four checks around every agent payment

Tvist runs four checks in a fixed order: jurisdiction agreed by Nash bargaining over 22 real regimes, consent enforced before settlement, escrow released only on delivery proof, and recall only where the regime allows it and the mandate was breached — plus law-cited disputes, an x402 pay-per-request rail, and machine-to-machine delegation. Implemented in this sandbox: 33 endpoints, 22 jurisdictions, 7 legal categories, and a passing test suite (pytest -q).

Why this exists

Five failure cases, and the check that stops each one

These are the failure cases when an AI agent pays over instant payment rails. Each card names the failure and the Tvist check that refuses it. The Run it button sends the failing request to this API and prints the refusal in the console (section Try live).

Case 1 · Irrevocability

Instant payments cannot be reversed by the sender

Pix, SEPA Instant, FedNow, and UPI settle in seconds and are final. If an agent pays the wrong party, the money is not recoverable afterwards. Check: POST /recall reverses a payment only where the region's rules allow recall and the payment breached the recorded consent.

Case 2 · No consent record

No stored record of what the human authorised

Without a stored mandate (budget, allowed merchants) there is no basis to refuse an out-of-scope payment before it settles, and no basis to recall it after. Check: POST /consent stores the mandate; POST /pay refuses any payment outside it with 403, before funds move.

Case 3 · Rule differences per rail

Each rail has different dispute rules

Pix MED allows a recovery request for 11 days. SEPA has the SCT Inst recall procedure. FedNow has no recall procedure. Reason codes are not portable between rails. Check: GET /regions lists all 22 regimes; POST /dispute accepts a reason code only if it is valid in the agreed region.

Case 4 · Prepayment risk

Paying before delivery on a final rail

An agent that prepays on an irrevocable rail has no remedy if the goods or service never arrive. Check: POST /escrow holds the funds; POST /escrow/{id}/release pays the payee only after the posted delivery proof equals the agreed condition.

Case 5 · Machine-to-machine authority

Delegation and agent-to-agent trade are unenforced

When an agent delegates spending to sub-agents, nothing enforces that a sub-agent stays within its delegator's budget. Two agents that have never interacted share no agreed jurisdiction. Check: POST /m2m/delegate refuses a delegation larger than its parent's budget; POST /m2m/handshake refuses a trade when no shared region exists.

Live · one POST /regions/recommend per pair

⚖️ Nash cooperation — 1–1, 1–n, n–n

Each mode draws buyer agents (left) and counterparties (right) and sends one real POST /regions/recommend call per pair, with the ranked preference lists shown in each box. A teal edge is labeled with the agreed region — the Nash bargaining solution over both rankings. A red dashed edge means the pair shares no region and must not transact. Agreed regions turn red on the globe above.

How the Nash choice works, step by step
  1. Rank. Each party lists the jurisdictions it accepts, best first. A list of length N gives its first choice N points, its second N−1, and so on. A region not on the list gets no points — it is unacceptable.
  2. Intersect. Only regions on both lists are feasible. If the intersection is empty, the answer is agreed_region: null: no deal, because any pick would force a jurisdiction on someone who rejected it.
  3. Multiply, don't add. For each feasible region, multiply the two scores: uclient × uagent. The region with the largest product wins. Multiplying is the point: a region one side loves and the other barely tolerates scores 3×1 = 3, while a region both rank second scores 2×2 = 4 — the product rewards mutual acceptability and punishes lopsided picks. This is the Nash bargaining solution, and neither party gains by demanding its own first choice: outside the agreed pick, the counterparty simply refuses.
  4. Break ties fairly. Equal products → prefer the region whose worse-off party scores higher (maximin), then the higher total, then the alphabetically first id — so the result is always deterministic.

Worked example (the 1–1 mode below): client ranks eu_sepa › br_pix › in_upi (3, 2, 1 points), agent ranks in_upi › br_pix › eu_sepa (3, 2, 1). Products: eu_sepa 3×1=3 · br_pix 2×2=4 · in_upi 1×3=3. Result: br_pix — each side's second choice beats either side's first, because it is the pick both can live with.

Key components

Seven components, one API — each card's button runs it live

Component

🗺️ Region recommender

POST /regions/recommend — reads each party's ranked list as ordinal utilities and returns the Nash bargaining solution: the region maximising uclient·uagent, ties broken by maximin. Example: client ranks SEPA first, agent ranks UPI first, both rank Pix second — the result is Pix. agreed_region: null means no shared region: do not transact.

Component

🔏 Consent vault

POST /consent stores the principal's budget and optional merchant allowlist; POST /pay enforces it. A payment above the budget or to a merchant not on the allowlist is refused with 403 before funds move.

Component

🔒 Programmable escrow

POST /escrow → /deliver → /release. Funds leave the payer at open, reach the payee only when proof == condition_expected. Contest blocks release; refund makes the payer whole.

Component

↩️ Region-aware recall

POST /recall — reverses a settled payment only when the region permits recall and the payment provably breached the mandate. FedNow refuses; UPI/Pix honour a breach.

Component

⚖️ Dispute taxonomy

POST /dispute — accepts a case only if the reason code is valid in the agreed region. Example: pix_med_return is rejected under in_upi. GET /taxonomy maps every reason code to one of 7 legal categories.

Component

🤖⇄🤖 Machine-to-machine trade

POST /m2m/delegate: an agent delegates part of its budget to a sub-agent. A delegation larger than its parent's budget is refused with 403; every chain traces to one root consent. POST /m2m/handshake: one call agrees the region, checks the mandate, and opens a funded escrow between two agents. Refusals (409 no shared region, 403 over mandate) happen before funds move.

Component

⚡ x402 on-rail payments

GET /x402/resource/… — pay-per-request over HTTP: the first request returns 402 Payment Required with the price and payee; the retry carries a signed X-PAYMENT header and returns the resource plus an X-PAYMENT-RESPONSE receipt. Each nonce settles once (replays refused). If the payment names a consent_id, the budget is checked before settlement.

Legal taxonomy · live from GET /taxonomy

Every dispute reason code maps to civil & commercial law

Each reason code maps to one of 7 legal categories, each with its civil-law and common-law basis. Each of the 22 jurisdictions lists its statutes, scheme rulebooks, and regulator, with a link to the official source for each instrument. An accepted POST /dispute returns the citation string in the response (legal_basis.citation); the full per-region list is at GET /regions/{region}/legal.

Personas

Four callers — each button runs that caller's requests

Persona

🛒 The shopping / booking agent

Job: spends its principal's money on flights, goods, subscriptions.
Without Tvist: prepays on final rails, overspends its mandate, no recourse.
With Tvist: escrows the fare, pays inside consent, disputes with valid codes.

Persona

🧑‍🦱 The principal (human)

Job: delegates — "book it, up to 500".
Without Tvist: discovers the 900-credit charge after settlement, on an irrevocable rail.
With Tvist: the 900 charge is a 403; a slipped-through breach is recalled.

Persona

🏪 The service-provider agent

Job: sells to agents it has never interacted with.
Without Tvist: either ships before payment and risks non-payment, or requires prepayment and the buyer carries the risk.
With Tvist: confirms funds are held in escrow before shipping; is paid when the delivery proof is accepted.

Persona

🏛️ The marketplace / platform

Job: hosts agent↔agent trades across borders.
Without Tvist: must implement dispute rules separately for each rail, corridor, and regulator.
With Tvist: one call returns the agreed jurisdiction and its full dispute regime.

Use case

🤝 Cross-border agent trade

EU client-agent, Brazilian seller-agent: the recommender lands on Pix as the Nash optimum, and both sides know the recall window before a credit moves.

Use case

🛰️ Marketplace dispute rail

Platforms route every contested trade through /dispute with region-valid codes — one taxonomy instead of N rail-specific ones.

Use case

🧾 Subscription guardrails

A recurring agent payment carries a standing consent; the first renewal above budget is refused, not refunded-after-the-fact.

Use case

🤖⇄🤖 Agent-swarm procurement

An orchestrator agent delegates bounded budgets to sub-agents. Each sub-agent sets up a trade with a seller agent in one call (region + mandate + funded escrow) and completes it on delivery proof. No human is involved in any step.

Use case

⚡ Pay-per-call agent APIs

An agent buys a paid resource over x402 — 402 challenge, signed X-PAYMENT, receipt — with its principal's consent cap enforced even on the irrevocable rail.

# Nash-optimal jurisdiction for two parties with different rankings  
curl -s -X POST $BASE/regions/recommend -H 'content-type: application/json' \
  -d '{"client_prefs":["eu_sepa","br_pix","in_upi"],"agent_prefs":["in_upi","br_pix","eu_sepa"]}'
→ {"agreed_region":"br_pix","method":"nash_bargaining","naive_client_first":"eu_sepa"}

# over-consent charge — refused before funds move  
curl -s -X POST $BASE/pay -d '{"ref":"p2","from_account":"alice","to_account":"shop",
  "amount":900,"region":"br_pix","consent_id":"c1"}'
→ 403 {"error":"payment 900 to shop exceeds consent c1"}

# escrow release before delivery — refused  
curl -s -X POST $BASE/escrow/e1/release
→ 403 {"error":"release condition not satisfied (not delivered)"}

# mandate-breach recall under UPI — allowed, payment reversed  
curl -s -X POST $BASE/recall -d '{"ref":"u1","consent_id":"cb"}'
→ {"reversed":true,"reason":"mandate breach recalled"}
